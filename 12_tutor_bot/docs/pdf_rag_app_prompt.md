# Промпт: новое приложение PDF RAG

Создай новый самостоятельный Python-проект: Streamlit-приложение с RAG, которое принимает PDF-документы, преобразует их в Markdown, сохраняет извлечённые изображения и позволяет задавать вопросы по содержимому загруженных документов.

Не изменяй никакой существующий проект и не используй его файлы напрямую. Его можно воспринимать только как ориентир по простой структуре: отдельный Streamlit UI, сервисы для документов и RAG, настройки в одном файле, рабочие данные в `data/`.

## Стек

- Python 3.11+
- Streamlit
- PyMuPDF (`pymupdf`) для чтения PDF, извлечения текста и встроенных изображений
- LangChain
- Ollama:
  - чат-модель через `OLLAMA_CHAT_MODEL`;
  - embedding-модель через `OLLAMA_EMBEDDING_MODEL`
- ChromaDB как локальная векторная база
- python-dotenv
- Langfuse — необязательная observability-интеграция, включается ключами в `.env`

## Основной сценарий

1. Пользователь загружает один или несколько PDF через Streamlit.
2. Приложение сохраняет исходный PDF.
3. Для каждого PDF приложение:
   - извлекает текст постранично;
   - извлекает встроенные изображения;
   - сохраняет изображения на диск;
   - создаёт Markdown-файл;
   - вставляет в Markdown относительные ссылки на извлечённые изображения близко к их исходному расположению на странице;
   - добавляет заголовок `## Страница N` для каждой страницы.
4. Приложение создаёт или обновляет Chroma-индекс по итоговым Markdown-файлам.
5. Пользователь задаёт вопросы в чате, а RAG отвечает только на основании загруженных документов.
6. Интерфейс показывает ответ и использованные фрагменты Markdown.

## Требования к хранению одного PDF

```text
data/documents/
  lecture.pdf
  lecture.md
  lecture/
    images/
      page-001-image-001.png
      page-001-image-002.jpg
      page-002-image-001.png
```

Пример содержимого `lecture.md`:

```md
# lecture

## Страница 1

Текст, извлечённый с первой страницы.

![Изображение 1 со страницы 1](lecture/images/page-001-image-001.png)

Продолжение текста.

## Страница 2

Текст второй страницы.

![Изображение 1 со страницы 2](lecture/images/page-002-image-001.jpg)
```

## Требования к PDF-конвертации

- Используй PyMuPDF.
- Сохраняй исходный формат изображения, если он поддерживается браузерами. Если формат неудобен для отображения, например CMYK или другой неподдерживаемый формат, конвертируй в PNG.
- Изображения должны иметь предсказуемые имена: `page-XXX-image-YYY.<ext>`.
- Не создавай битые ссылки на изображения.
- Не извлекай один и тот же объект изображения несколько раз в пределах одного PDF.
- Максимально сохраняй порядок: текстовые блоки и изображения на странице должны сортироваться по координате сверху вниз.
- Если страница — скан без извлекаемого текста, не реализуй OCR. Добавь в Markdown заметку: `> На странице не обнаружен текстовый слой. Для распознавания требуется OCR.`
- Подготовь `pdf_service.py` так, чтобы OCR можно было добавить позже без переписывания UI и RAG.
- При повторной загрузке PDF с тем же именем перезаписывай связанный Markdown и папку изображений, удаляя устаревшие файлы.
- При очистке документов удаляй и PDF, и сгенерированные `.md`, и папки с изображениями, и Chroma-индекс.

## Требования к RAG

- Индексируй только `.md`-файлы.
- Используй `MarkdownHeaderTextSplitter`, затем `RecursiveCharacterTextSplitter`.
- Сохраняй в метаданных имя исходного Markdown-файла, заголовок раздела и идентификатор чанка.
- Ответ должен быть на русском языке и опираться только на найденный контекст.
- Если контекста недостаточно, ответ должен быть: `В загруженных документах нет информации для ответа на этот вопрос.`
- Показывай пользователю список найденных источников: заголовок, релевантность и текст фрагмента.
- Не индексируй бинарные файлы изображений.

## Требования к Streamlit UI

- Заголовок приложения: `PDF RAG: PDF → Markdown`.
- В боковой панели:
  - настройки используемых моделей Ollama;
  - статус индекса;
  - загрузка нескольких PDF;
  - кнопка `Сохранить и обновить индекс`;
  - кнопка `Очистить документы`;
  - перечень загруженных PDF;
  - количество страниц и изображений после конвертации;
  - раскрывающийся предпросмотр Markdown для каждого конвертированного PDF.
- В основной части:
  - чат с историей сообщений в `st.session_state`;
  - блоки с источниками под каждым ответом.
- Ошибки показывай через `st.error`.
- Во время конвертации и индексации используй `st.spinner`.

## Langfuse

- Интеграция должна быть необязательной: если ключей нет, приложение продолжает работать.
- Если ключи есть, записывай события:
  - `pdf_converted`;
  - `pdf_conversion_failed`;
  - `index_created`;
  - `answer_question`;
  - `retrieve_context`;
  - `generate_answer`.
- Для `pdf_converted` сохраняй имя файла, число страниц, число изображений и путь итогового Markdown.

## Требования к стилю кода

- Не добавляй комментарии в код.
- Не используй однострочные `if`.
- Оставляй пустую строку перед каждым `return`.
- Используй понятные имена переменных.
- Добавляй type hints.
- Не помещай бизнес-логику в `app.py`.
- Каждый модуль должен иметь одну понятную ответственность.
- Не используй облачные API для конвертации PDF или RAG.
- Не используй `print`.
- Не добавляй секреты в код.
- Не создавай файлы-примеры PDF, изображения, ассеты или бинарные файлы.
