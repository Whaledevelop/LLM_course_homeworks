Benchmark был запущен на 1200 документах командой:

```bash
python scripts/run_demo.py --rebuild --benchmark
```

| Профиль | Metric | search_ef | M | Semantic Recall@5 | Semantic MRR | Semantic latency, ms | Hybrid Recall@5 | Hybrid MRR | Hybrid latency, ms |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| cosine_fast | cosine | 20 | 12 | 1.000 | 1.000 | 3.13 | 1.000 | 1.000 | 78.54 |
| cosine_balanced | cosine | 50 | 16 | 0.857 | 0.857 | 2.63 | 1.000 | 0.929 | 81.09 |

На этом синтетическом датасете все профили дают высокий Recall@5. Hybrid search стабилизирует качество, но работает медленнее, потому что дополнительно считает keyword score по документам. На больших данных lexical part нужно выносить в отдельный BM25-индекс, иначе latency будет расти линейно.

## ANN-алгоритм

В ChromaDB используется HNSW. В отличие от IVF, HNSW строит граф связей между ближайшими векторами и при поиске проходит по графу к наиболее похожим точкам. Это хорошо подходит для интерактивного RAG, где важна низкая задержка.

Сравнение с алгоритмами из лекции:

| Алгоритм | Идея | Плюсы | Минусы |
| --- | --- | --- | --- |
| HNSW | Граф ближайших соседей | Высокий recall, быстрая выдача, хорошо для динамических коллекций | Больше расход RAM |
| IVF | Деление пространства на кластеры | Быстро на больших объемах, меньше проверяемых векторов | Нужно обучать центроиды, качество зависит от `n_probe` |
| ANNOY | Набор random projection trees | Простая реализация, хорош для статичных индексов | Хуже с частыми обновлениями и сложной фильтрацией |

Так как Chroma использует HNSW, в проекте сравниваются не IVF/ANNOY, а доступные параметры HNSW и similarity metrics.

## Параметры Chroma/HNSW

В benchmark сравниваются профили:

- `cosine_fast`: `search_ef=20`, `M=12`.
- `cosine_balanced`: `search_ef=50`, `M=16`.
- `cosine_accurate`: `search_ef=100`, `M=32`.
- `l2_balanced`: `space=l2`.
- `ip_balanced`: `space=ip`.

Ожидаемые trade-offs:

- Увеличение `search_ef` обычно повышает recall, но увеличивает latency.
- Увеличение `M` повышает связность графа и качество поиска, но увеличивает память и время индексации.
- `cosine` удобен для нормализованных текстовых эмбеддингов.
- `l2` полезен, когда важна евклидова геометрия.
- `ip` полезен для embeddings, обученных под dot product.

## Semantic vs Hybrid

Semantic search хорошо работает с переформулировками, потому что ищет близость смысловых представлений. Keyword scoring лучше ловит точные термины: `HNSW`, `IVF`, `BM25`, `Recall@K`, названия баз и версий.

Hybrid search объединяет оба сигнала через Reciprocal Rank Fusion. В RAG это полезно, потому что пользовательские вопросы часто содержат и смысловое описание, и точные технические маркеры.

## Метаданные

Фильтрация по метаданным нужна, когда область поиска заранее ограничена:

- только документы категории `ann_algorithm`;
- только источник `lecture_notes`;
- только материалы за конкретный год;
- только документы определенного уровня сложности.

Это снижает количество нерелевантного контекста, ускоряет генерацию ответа и уменьшает риск галлюцинаций.

## Интерпретация метрик

`Recall@K` показывает, попал ли хотя бы один релевантный документ в top-k. `MRR` дополнительно учитывает позицию первого релевантного результата. Если Recall@5 высокий, но MRR низкий, значит нужные документы находятся, но не на первых позициях.

Latency нужно смотреть вместе с качеством. Самый быстрый профиль не всегда лучший для RAG: если он возвращает нерелевантный контекст, генерация будет хуже даже при низкой задержке retrieval.

## Вывод

Для учебного RAG оптимальным профилем является `cosine_balanced`: он использует cosine distance, умеренный `search_ef` и стандартный размер графа. `cosine_fast` полезен для демонстрации минимальной задержки, а `cosine_accurate` - для проверки, насколько качество улучшается при более дорогом поиске.

ChromaDB закрывает обязательную часть домашки: настройка БД, индексация, ANN-параметры, semantic search, similarity metrics, metadata filtering, top-k и end-to-end RAG. Бонусная часть закрыта через датасет 1000+ документов, hybrid search, batch benchmark и сравнение нескольких ANN-настроек.
