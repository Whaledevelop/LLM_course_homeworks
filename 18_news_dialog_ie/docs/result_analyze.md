# Анализ результатов

Benchmark запускается командой:

```powershell
python scripts/run_demo.py --sample-size 100 --batch-size 16 --rebuild-cache
```

На машине без локальной 7B-модели проверяется профиль `rules`. Он нужен как быстрый baseline и как контрольный пайплайн, который работает без GPU.

Фактический быстрый запуск в текущем окружении:

| extractor | examples | batch_size | docs_per_second | estimated_tokens_per_second | mean_latency_ms | peak_rss_mb | precision | recall | f1 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| rules | 100 | 16 | 10311.19 | 588769.05 | 0.10 | 0.0 | 0.913 | 1.000 | 0.955 |

Фактические значения сохраняются в `data/benchmark_results.csv`.

Дополнительно сохраняются `data/gold_annotations.csv` и `data/extraction_predictions.csv`. Эти CSV удобны для ручной проверки false positive/false negative и для импорта в notebook.

## Качество

Rule-based baseline хорошо извлекает:

- новостные источники: `Reuters`, `BBC`, `CNN`, `AP News`;
- даты в формате `Month DD, YYYY`;
- часто встречающиеся организации;
- заранее известные типы событий: `meeting`, `earthquake`, `investigation`, `layoffs`;
- часть impact-фраз, если они явно сформулированы.

Слабые места baseline:

- новые организации вне словаря не извлекаются;
- `PERSON` извлекается регулярным выражением и может захватывать лишние title-слова;
- `IMPACT` зависит от формулировки;
- сложные отношения между несколькими событиями требуют LLM или отдельной модели relation extraction.

spaCy закрывает часть слабых мест по `PERSON/ORG/LOC/DATE`, но не решает `IMPACT` и специфичные новостные события. Поэтому в проекте spaCy объединяется с rule-based IE.

## Производительность

Самый быстрый режим - `rules`: он почти не потребляет память и хорошо подходит для потоковой предразметки больших датасетов. spaCy медленнее, но дает лучшее покрытие сущностей. LLM-профиль самый дорогой, зато лучше переносится на новые типы событий.

Ожидаемые trade-offs:

| Профиль | Скорость | Качество | Память | Когда использовать |
| --- | --- | --- | --- | --- |
| rules | высокая | среднее | низкая | фильтрация, baseline, weak labels |
| spaCy + rules | средняя | выше baseline | средняя | NER на CPU |
| LLM fp16 | низкая/средняя | высокое | высокая VRAM | сложные события и отношения |
| LLM int8 | средняя | немного ниже fp16 | ниже VRAM | GPU с ограниченной памятью |

## Оптимизация

В проекте реализованы обязательные оптимизации:

- потоковая загрузка WildChat-1M;
- batch inference;
- cache результатов по профилю;
- замер throughput и latency;
- отдельный quantized-флаг для LLM;
- разделение извлечения, оценки и UI.

Для production-версии следующими шагами были бы:

- заменить регулярные словари на gazetteer из Wikidata/GDELT;
- добавить LLM-as-judge или ручную разметку 200-500 примеров;
- использовать structured generation с JSON grammar;
- хранить результаты в DuckDB/PostgreSQL;
- визуализировать события по источникам, датам и локациям в BI.
